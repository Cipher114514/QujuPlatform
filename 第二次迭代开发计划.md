### M1 — P1：活动评价与复盘（US-021/023）| 10h

> **Git 分支**: `feature/m1-activity-review`

#### 覆盖的用户故事

| 编号 | 故事 | 验收要点 |
|------|------|------|
| US-021 | 活动复盘数据统计与图文花絮 | 参与人数/签到率/缺席率看板，多图+文字花絮上传 |
| US-023 | 参与者星级活动评价流 | 1-5 星评分 + 文本评价，关联活动详情页 |

#### 验收标准

**US-021 活动复盘数据统计与图文花絮**：

- [ ] 活动结束后，发起者可在复盘页查看：报名总人数、实际签到人数、缺席人数、签到率百分比
- [ ] 签到率以环形图或进度条形式直观展示（纯 CSS 实现，不依赖第三方图表库）
- [ ] 发起者可上传多张活动花絮图片（单张 ≤ 10MB）并配文字描述
- [ ] 花絮图片以网格墙形式展示，支持点击放大预览
- [ ] 边缘情况：无签到记录时，签到率显示为 0%，提示"暂无签到数据"
- [ ] 边缘情况：仅有报名但无人签到时，数据看板正常展示，签到率 0%
- [ ] 仅活动发起者可编辑花絮，普通参与者只读

**US-023 参与者星级活动评价流**：

- [ ] 活动结束后，已签到参与者可对活动进行 1-5 星评分
- [ ] 评价表单包含：星级选择（点击第 N 颗星选中 1-N 星）、文本评价（选填，限 500 字）
- [ ] 提交后评价实时展示在活动详情页的评价列表区
- [ ] 评价列表按时间倒序排列，每条显示：用户名、星级、评价内容、精确到分钟的时间戳
- [ ] 边缘情况：未签到用户访问评价页时隐藏评价表单，提示"签到后方可评价"
- [ ] 边缘情况：同一用户对同一活动仅可评价一次，重复提交时后端拦截并提示"您已评价过该活动"
- [ ] 边缘情况：输入框拦截空提交（纯空格视为空），超出 500 字符给出提示
- [ ] 活动平均评分在详情页顶部展示（如 ★ 4.3 · 12条评价）

#### 新增实体

`Review`（评价）、`Retrospect`（复盘）、`ActivityGallery`（花絮图片）

#### 后端接口

```
# 评价
POST   /activities/:id/review        { rating(1-5), content }
GET    /activities/:id/reviews       ?page=&size=
GET    /activities/:id/review/avg     → { avgRating, totalCount }

# 复盘
GET    /activities/:id/retrospect     → { registrations, checkins, absentees, rate }
POST   /activities/:id/retrospect/gallery  { images[], description }
GET    /activities/:id/retrospect/gallery  ?page=&size=
```

#### 前端页面

- **`#/activity/:id/review`**：星级组件 + 文本输入 + 已有评价列表
- **`#/activity/:id/retrospect`**：数据卡片（签到率环形图）+ 花絮图片墙 + 上传按钮

### M2 — P2：AI 内容安全（US-010/011-B）| 8h

> **Git 分支**: `feature/m2-ai-content-safety`

#### 覆盖的用户故事

| 编号 | 故事 | 验收要点 |
|------|------|------|
| US-010 | AI 敏感关键词内容前置过滤 | 活动发布/留言提交时调用 AI 敏感词过滤，拦截违规内容 |
| US-011-B | AI 图片多标签自动分类 | 上传图片时调用 AI 视觉接口，自动打 3+ 个标签存入数据库 |

#### 验收标准

**US-010 AI 敏感关键词内容前置过滤**：

- [ ] 活动发布时，标题和描述文本提交至 AI 敏感词过滤接口，返回通过/拦截结果
- [ ] 留言提交时，留言内容同样经过 AI 敏感词过滤
- [ ] 被拦截的请求返回统一错误码 `451`，前端展示"内容包含违规信息，请修改后重试"
- [ ] AI API 超时（3 秒）或不可用时，自动降级为本地关键词黑名单过滤
- [ ] 本地黑名单至少包含：常见政治敏感词、违法广告词、人身攻击词，不少于 20 条
- [ ] 边缘情况：空文本直接放行，不调用 AI 接口
- [ ] 边缘情况：纯英文或中英混合文本也可正确过滤
- [ ] 过滤日志记录（时间、用户ID、被拦截文本摘要、拦截原因），不记录完整原文以保护隐私

**US-011-B AI 图片多标签自动分类**：

- [ ] 用户上传活动配图或头像后，异步调用 AI 接口进行图片标签识别
- [ ] 每张图片至少返回 3 个标签（如：户外、运动、人群）
- [ ] 标签数据随图片记录存入数据库，供活动搜索检索
- [ ] 若 AI 视觉接口不可用（qwen 不支持图片），降级为基于文件名+上传分类的规则标签
- [ ] 边缘情况：图片格式不支持时，返回空标签列表，不报错阻塞上传流程
- [ ] 边缘情况：标签识别超时 5 秒自动放弃，不影响图片正常上传

#### 后端接口

```
POST /ai/content-filter     { text }  → { passed: bool, reason?: string }
POST /ai/image-tags         { imageUrl } → { tags: string[] }
```

#### 埋点位置

- `ActivityService.createActivity()` 提交前调用 filter
- `CommentService.addComment()` 提交前调用 filter
- `UploadController` 上传成功后异步调用 image-tags

#### 验收标准

- [ ] 接入高德地图 JS API 2.0，页面加载后渲染全屏地图视图
- [ ] 根据当前地图视野范围（bounds），自动请求该区域内的活动数据

### M4 — P4：动态二维码签到核销（US-019）| 10h

> **Git 分支**: `feature/m4-activity-checkin`

#### 覆盖的用户故事

| 编号 | 故事 | 验收要点 |
|------|------|------|
| US-019 | 动态二维码现场签到与核销 | 定时刷新 30s 动态码，发起者扫码核销，拒绝过期码 |

#### 验收标准

- [ ] 参与者打开签到页，展示定时刷新的动态二维码（每 30 秒自动刷新）
- [ ] 二维码内容使用 JWT 签名（payload: `{activityId, userId, exp}`），防止伪造
- [ ] 发起者打开核销页，调用摄像头扫描参与者出示的二维码
- [ ] 扫码成功解析后，调用后端核销接口校验 token 有效性
- [ ] 核销成功后，参与者签到状态变更为"已签到"，前端显示成功动画
- [ ] 二维码过期（超过 30 秒）时核销接口返回"二维码已过期，请刷新后重试"
- [ ] 边缘情况：同一二维码重复扫描时返回"该用户已签到"，不报错
- [ ] 边缘情况：非活动参与者出示的二维码，核销时返回"该用户未报名此活动"
- [ ] 边缘情况：摄像头权限被拒绝时，核销页降级为手动输入验证码模式（6 位数字，同样 30 秒刷新）
- [ ] 发起者可查看签到名单：已签到/未签到人员列表
- [ ] 仅活动发起者可进入核销页，其他用户访问时提示"仅活动发起者可核销"

#### 后端接口

```
GET  /activities/:id/qrcode          → { token, expireAt }       获取当前有效二维码
POST /activities/:id/checkin         { token }                    核销签到
GET  /activities/:id/checkin/status                              当前用户签到状态
GET  /activities/:id/checkin/list                                 签到名单（仅发起者可见）
```

#### 前端页面

- **`#/activity/:id/qrcode`**（参与者端）：动态二维码 + 倒计时进度条 + 活动名称
- **`#/activity/:id/checkin`**（发起者端）：摄像头扫码区域 + 手动输入验证码降级模式 + 签到名单

#### 覆盖的用户故事

| 编号 | 故事 | 验收要点 |
|------|------|------|
| US-029 | 小队内群聊空间与队内专属活动 | 小队自动生成群聊，成员进退同步，队长可发队内活动 |

#### 验收标准

- [ ] 小队创建成功后，自动生成对应的群聊会话
- [ ] 新成员加入小队时自动加入群聊，退出小队时自动离开群聊
- [ ] 群聊支持发送文字消息，消息实时推送给所有在线成员
- [ ] 群聊消息通过 WebSocket `/topic/team/{teamId}` 广播，复用现有 STOMP 基础设施
- [ ] 群聊历史消息支持分页加载，最新消息显示在底部
- [ ] 消息气泡显示：发送者昵称 + 头像 + 内容 + 发送时间
- [ ] 队长可在小队内发布"队内专属活动"，Activity 实体新增 `teamId` 字段标记
- [ ] 队内活动仅小队成员可见，不出现在公开活动信息流中
- [ ] 边缘情况：用户退出小队后，无法再查看该小队群聊历史
- [ ] 边缘情况：空消息拦截，不能发送纯空格内容
- [ ] 边缘情况：WebSocket 不可用时，降级为每 3 秒 REST 轮询

#### 后端接口

```
GET    /teams/:id/messages                ?page=&size=       群聊消息历史
POST   /teams/:id/messages                { content }        发送群聊消息
WebSocket: /topic/team/{teamId}                              群聊实时推送

POST   /teams/:id/activities              { ...活动字段 }     创建队内活动
GET    /teams/:id/activities              ?page=&size=        队内活动列表
```

#### 前端页面

- **`#/team/:id/chat`**：群聊窗口 — 消息气泡列表 + 底部输入框 + 发送按钮 + 消息时间戳

#### 验收标准

## 六、开发规范

1. **每个人只修改自己负责的文件**，不碰别人的 JS/CSS/Java
2. **模块间仅通过 `<a href="#/xxx">` 跳转**，不注入 DOM
3. **API 契约优先**：接口格式参照现有规范，响应用 `Result<T>` 包装
4. **分支命名**：`feature/m{N}-{模块名}`，每日至少 push 一次
5. **Mock 先行**：每个页面内置 `USE_MOCK` 开关，后端未好也能开发
6. **新增实体需同步 DataInitializer** 的测试数据
