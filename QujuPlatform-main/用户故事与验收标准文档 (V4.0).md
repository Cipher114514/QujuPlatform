# 用户故事与验收标准文档 (V4.0 )

本文档详尽规定了“线下活动社交平台（趣聚）”系统全量用户故事的业务描述、优先级、工时评估及具体的单体验收标准（Acceptance Criteria）。

## 一、 迭代一：MVP 纯文本闭环底座（共 13 个用户故事）

### US-001: 个人用户基础注册（去依赖激活）

- **用户故事描述**：作为一个普通游客，我希望能够填写基本信息注册账号，以便后续能够登录系统。
- **验收标准**：
  - [ ] 必须支持手机号/邮箱作为唯一标识符进行注册，密码需满足复杂度校验（如包含字母与数字，至少6位）。
  - [ ] 注册接口应在无须后台邮件或短信验证码激活的前提下直接落库生效（状态默认为可用态）。
  - [ ] 边缘情况：重复手机号/邮箱注册需被拦截，并给出明确的“账号已存在”提示。

### US-002: 商家用户身份注册（默认可用态）

- **用户故事描述**：作为一个新商家用户，我希望能够填写营业执照等基本信息注册商家账号，以便后续能够入驻平台发布活动。
- **验收标准**：
  - [ ] 提供商家注册表单，包含统一社会信用代码/营业执照上传字段。
  - [ ] 注册落库后默认即可用状态，供后续提交后台人工审核追溯。
  - [ ] 校验同一社会信用代码不可重复注册。

### US-003: JWT统一认证登录机制（鉴权大闸）

- **用户故事描述**：作为一个已注册的个人或商家用户，我希望能够通过账号密码进行登录，以便获取系统访问令牌。
- **验收标准**：
  - [ ] 前端提供邮箱/手机号与密码输入框，后端验证通过后下发 JWT Token。
  - [ ] 后续所有需要权限的接口必须在 Header 中携带此 Token 进行合法性校验。
  - [ ] 密码错误次数超限需触发临时锁定机制。

### US-004: 个人资料与兴趣标签轻量化维护

- **用户故事描述**：作为一个个人用户，我希望能够维护我的个人信息和兴趣标签，以便向他人展示自我。
- **验收标准**：
  - [ ] 支持修改头像、昵称（全平台唯一）、性别、生日、个性签名。
  - [ ] 支持增删改个人兴趣标签（如：剧本杀、飞盘、徒步等）。
  - [ ] 数据更新后实时落库并在个人主页生效。

### US-005: 商家资料与品牌服务领域维护

- **用户故事描述**：作为一个商家用户，我希望维护店铺的品牌和服务领域，以便精准吸引用户。
- **验收标准**：
  - [ ] 支持维护商家名称、品牌昵称、服务领域描述等字段。
  - [ ] 修改信息需通过表单校验并成功持久化至数据库。

### US-006: 基础纯文本活动发布（剔除地图）

- **用户故事描述**：作为一个已登录的商家或个人用户，我希望能够发布一场不包含地理位置的纯文本活动，以便招募参与者。
- **验收标准**：
  - [ ] 必填字段校验：活动标题、活动内容描述、开始/结束时间、最大参与人数限制。
  - [ ] 提交成功后，活动状态应直接流转为“发布成功”。
  - [ ] 边界：活动结束时间必须大于开始时间，且开始时间不得早于当前系统时间。

### US-008: 活动全量草稿箱暂存与重新编辑

- **用户故事描述**：作为一个活动发布者，我希望能够将未编辑完的活动暂存至草稿箱，以便后续随时调取并重新编辑发布。
- **验收标准**：
  - [ ] 提供“存为草稿”功能，状态标记为草稿态，不进入首页活动信息流。
  - [ ] 在草稿箱列表中点击编辑，可加载原填写的文本内容，并支持全量要素修改与再次提交。

### US-009: 纯文本活动修改与发布撤销流

- **用户故事描述**：作为一个活动发布者，我希望能够修改已发布的活动或撤销发布，以便应对临时变动。
- **验收标准**：
  - [ ] 允许在报名开始前修改已发布的活动详情（人数限制、起止时间等）。
  - [ ] 提供“撤销发布”按钮，撤销后活动在前台下架并转为草稿态或已取消态。

### US-012: 首页活动信息流瀑布卡片展示

- **用户故事描述**：作为一个游客或用户，我希望打开首页时能以瀑布流卡片的形式浏览平台发布的活动，以便快速发现感兴趣的内容。
- **验收标准**：
  - [ ] 页面以卡片列表分页加载活动信息。
  - [ ] 卡片需包含活动标题、简介、时间、发起人等核心摘要要素。
  - [ ] 翻页加载流畅，无明显卡顿。

### US-013: 模糊多条件关键词文本搜索活动

- **用户故事描述**：作为一个用户，我希望通过关键词模糊搜索活动，以便快速找到特定活动。
- **验收标准**：
  - [ ] 提供搜索输入框，支持根据活动标题或简介进行关键词匹配。
  - [ ] 支持多条件组合筛选（如按时间段、按活动标签分类）。
  - [ ] 无匹配结果时给出“未找到相关活动”的提示。

### US-015: 活动详情页全要素纯文本多维渲染

- **用户故事描述**：作为一个用户，我希望点击卡片后能进入详情页查看活动的完整纯文本内容，以便决定是否参与。
- **验收标准**：
  - [ ] 页面完整渲染标题、详细描述、起止时间、人数上限、当前报名人数等信息。
  - [ ] 排版清晰，纯文本多维信息展示无乱码或缺失。

### US-043: 平台公共基础留言板与单向回复

- **用户故事描述**：作为一个注册用户，我希望能够在活动或公共区域留言，并针对单条留言进行回复，以便进行线上互动。
- **验收标准**：
  - [ ] 留言内容支持纯文本输入，展示发布者的用户名及精确到秒的时间戳。
  - [ ] 留言与回复呈现单向树形结构展示（主留言 -> 首次回复）。
  - [ ] 边缘约束：输入框拦截为空提交，超出200字符限制需给出提示。

### US-044: 留言板不良言论举报与状态标记

- **用户故事描述**：作为一个用户，我希望能够对留言板内的恶意或违规言论进行举报，以便维护社区健康。
- **验收标准**：
  - [ ] 任意登录用户可点击某条留言的“举报”按钮，需记录举报原因枚举（垃圾广告、人身攻击等）。
  - [ ] 举报成功后，该留言在当前视图的状态应标记为“已举报/审核中”，防止二次扩散。

## 二、 迭代二：空间、核心社交、AI 演进与质量加固（共 20 个用户故事）

### US-014: 地图模式查看活动分布（高德SDK）

- **用户故事描述**：作为一个用户，我希望能在地图上直观查看到附近的线下活动分布，以便根据地理位置就近参与。
- **验收标准**：
  - [ ] 接入高德地图 SDK，在前端页面渲染出带有标记点（Marker）的地图视图。
  - [ ] 点击标记点需能轻量化弹出该活动的摘要信息卡片，并可点击跳转至活动详情页。
  - [ ] 异常处理：在未授权定位或网络延迟时，应降级展示或提示加载失败。

### US-016: 基础活动报名与截止前自主取消

- **用户故事描述**：作为一个用户，我希望能够报名感兴趣的活动，并在报名截止前取消，以便灵活安排行程。
- **验收标准**：
  - [ ] 点击报名扣减活动剩余名额，状态机流转为“已报名”。
  - [ ] 在活动截止时间前，用户可点击“取消报名”，名额释放。
  - [ ] 超过活动报名截止时间，取消按钮置灰或隐藏。

### US-018: 报名满额排队等待队列（FIFO）

- **用户故事描述**：当活动报名人数已满时，我希望能够排队候补，以便在有人取消时按先后顺序替补。
- **验收标准**：
  - [ ] 采用先进先出（FIFO）队列算法，记录超出名额限制的报名用户先后顺序。
  - [ ] 当有正式名额被用户取消释放时，队列首位用户状态自动由“排队中”变更为“报名成功”。

### US-019: 动态二维码现场签到与核销机制

- **用户故事描述**：作为活动发起者/商家，我希望能够使用动态二维码对参与者进行现场核销签到，以便防范代签风险。
- **验收标准**：
  - [ ] 系统为每场活动生成定时刷新的动态核销二维码（有效期如30秒刷新一次）。
  - [ ] 参与者出示该动态码，发起者端扫描后校验通过，状态变更为“已签到/已核销”。
  - [ ] 边界：同一二维码或已失效/过期的二维码提示核销失败。

### US-021: 活动复盘数据统计与图文花絮

- **用户故事描述**：作为一个活动发起者，我希望在活动结束后查看复盘数据并上传图文花絮，以便沉淀活动成果。
- **验收标准**：
  - [ ] 记录活动的最终参与人数、签到率及缺席率等基础数据看板。
  - [ ] 允许发起者上传多张图片和文字总结作为活动花絮展示。

### US-023: 参与者星级活动评价流

- **用户故事描述**：作为一个参与者，我希望活动结束后能对该活动进行星级评价，以便提供真实的反馈。
- **验收标准**：
  - [ ] 提供 1-5 星的评分组件以及文本评价输入框。
  - [ ] 评价提交后关联至活动详情页或发起者主页展示。

### US-024: 深度好友关系网与单向关注升级

- **用户故事描述**：作为一个用户，我希望能够关注感兴趣的其他用户，以便建立社交关系网。
- **验收标准**：
  - [ ] 提供“关注”功能按钮，支持单向关注操作。
  - [ ] 个人中心需展示“我的关注”与“我的粉丝”列表数据统计。

### US-027: 兴趣小队基础创建与发现申请

- **用户故事描述**：作为一个用户，我希望能够创建兴趣小队或申请加入公开小队，以便聚集同好。
- **验收标准**：
  - [ ] 填写小队名称、简介、标签创建小队，创建者自动成为队长。
  - [ ] 提供小队发现列表，支持按名称/标签查找；公开小队可直接一键加入，审核小队需提交申请等待队长同意。

### US-029: 小队内群聊空间与队内专属活动

- **用户故事描述**：作为一个小队成员，我希望拥有队内专属的群聊空间并能参与队内活动，以便进行深度交流。
- **验收标准**：
  - [ ] 小队创建后自动生成群聊，成员进退小队同步加群退群。
  - [ ] 队长/管理员可发布仅小队成员可见的队内专属活动。

### US-031: 小队共享相册与一键强制解散

- **用户故事描述**：作为一个队长/成员，我希望能够管理小队相册，并在必要时解散小队，以便维护小队生命周期。
- **验收标准**：
  - [ ] 成员可上传活动照片至小队共享相册，队长/管理员拥有删除相册内容的权限。
  - [ ] 提供“解散小队”功能（仅队长可用），解散后所有成员退出，群聊及队内活动停用。

### US-035: 基础双向私聊文字消息发送

- **用户故事描述**：作为一个用户，我希望能够给好友发送即时私聊文字消息，以便进行一对一沟通。
- **验收标准**：
  - [ ] 建立基于 WebSocket（或高频轮询兜底）的长连接，支持发送文字并在聊天框即时显示。
  - [ ] 简化逻辑：不包含已读状态与消息撤回，仅支持在线实时转发及落库存储。

### US-038: 管理后台登录与平台注册用户管训

- **用户故事描述**：作为一个平台总管理员，我希望通过独立的后台系统登录，并全面查看和管理全平台所有个人和商家的账号状态。
- **验收标准**：
  - [ ] 提供一套与前台完全分离的后台管理登录入口，需强校验超级管理员账户密码。
  - [ ] 进入后台用户管理大盘，能够分页检索所有注册用户，并提供一键“封禁/解封”按钮，封禁后前台登录强行拒绝。

### US-041: 违规内容下架与商家资质追溯审计

- **用户故事描述**：作为一个内容风控官，我希望能在后台一键下架违规活动，并对商家的营业执照资质进行人工审核，以便维护平台生态。
- **验收标准**：
  - [ ] 后台活动列表提供一键下架功能，下架后活动状态置为终止，前台信息流瞬间不可见。
  - [ ] 专设“商家审批看板”，集中展示商家注册时上传的营业执照凭证图片，提供通过/拒绝审批操作。

### US-007: [AI] 基于大模型的 AI 活动一键策划

- **用户故事描述**：作为活动发起者，我希望系统能根据输入的主题自动生成活动策划案，以便提高内容创作效率。
- **验收标准**：
  - [ ] 接入大模型 API，提供精简的 Prompt 模板（输入活动类型、受众、时长，输出策划大纲）。
  - [ ] 生成的内容可在前端富文本框中预览并进行二次修改保存。

### US-010: [AI] AI 敏感关键词内容前置过滤

- **用户故事描述**：作为平台方，我希望发布的内容在入库前能被AI自动识别并拦截敏感词，以保障社区合规。
- **验收标准**：
  - [ ] 在活动发布和留言提交环节，调用 AI 敏感词过滤接口。
  - [ ] 若文本包含违规关键词，直接拦截保存操作，并提示修改后重试。

### US-011-B: [AI] AI 图片多标签自动分类总结

- **用户故事描述**：作为系统，我希望上传的活动图片能被自动打上标签分类，以便辅助用户检索。
- **验收标准**：
  - [ ] 用户上传活动配图时，后台调用 AI 视觉接口识别图元特征（打上不少于3个标签）。
  - [ ] 标签数据随图片对象一并存入数据库，供全局检索功能调用。

### NFR-001: 核心交互接口高响应性能（亚秒级）

- **用户故事描述**：系统核心接口必须具备高响应性能，以保证用户体验。
- **验收标准**：首页信息流、活动搜索与详情加载等核心接口，在标准网络环境下 95% 的请求响应时间必须低于 1000ms（亚秒级）。

### NFR-002: 鉴权与敏感数据加密安全（数据防漏）

- **用户故事描述**：系统需保障数据传输与存储的安全性，防止信息泄露。
- **验收标准**：用户密码必须采用加盐哈希（如 bcrypt）存储，传输层采用 HTTPS 协议，核心接口必须校验 JWT 合法性。

### NFR-003: 抢单秒杀高并发与高可用（Redis锁）

- **用户故事描述**：在互斥抢票/报名场景下，系统需防范高并发导致的数据超卖。
- **验收标准**：在活动报名/抢票等核心互斥场景下，必须引入 Redis 分布式锁机制，确保在高并发请求下不出现超卖现象。

### NFR-004: 跨端响应式兼容与无缝容灾恢复

- **用户故事描述**：系统应具备良好的跨端适应能力及容灾防崩溃设计。
- **验收标准**：页面布局需兼容PC端及移动端主流分辨率；服务应具备无状态分布式设计，核心模块故障时能够快速降级或启用容灾备份节点。

## 三、 选做故事点详情（US-032 ~ US-042）

### US-032 (选做): 履约深度基础积分榜

- **用户故事描述**：作为一个小队成员，我希望系统能根据我的活跃度自动计算积分并展示排名，以便体现我的贡献度。
- **验收标准**：
  - [ ] 设置合理机制对每个小队成员进行积分统计（如参与小队活动、发布动态等）。
  - [ ] 根据积分生成队内排行榜，展示成员昵称、积分和名次。

### US-033 (选做): 小队内管理员权限授予与撤销

- **用户故事描述**：作为一个小队队长，我希望能够指定或撤销小队管理员，以便协助管理小队。
- **验收标准**：
  - [ ] 队长可在成员列表中对普通成员进行“设为管理员”或“撤销管理员”操作。
  - [ ] 管理员获得协助审核成员、管理公告、活动等子权限。

### US-036 (选做): 私聊消息单条已读状态（ACK读报文握手机制）

- **用户故事描述**：作为一个私聊发送方，我希望能够看到对方是否已读取我发送的消息，以便确认沟通进度。
- **验收标准**：
  - [ ] 在私聊对话界面，针对单条消息增加“已读/未读”状态回执（ACK报文机制）。
  - [ ] 对方点击查看该对话时，状态实时变更为已读。

### US-037 (选做): 发送后2分钟内消息撤回

- **用户故事描述**：作为一个消息发送方，我希望能够撤回发出不久的错误消息，以便挽回尴尬局面。
- **验收标准**：
  - [ ] 消息发送后 2 分钟内，长按或点击该消息可触发“撤回”操作。
