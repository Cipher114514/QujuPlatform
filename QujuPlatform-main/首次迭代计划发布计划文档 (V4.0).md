# 首次迭代计划发布计划文档 (V4.0)

## 一、 文档说明

本文档为“线下活动社交平台（趣聚）”综合实践项目的版本发布与迭代计划。针对实践周时间周期（迭代一 2.5天，迭代二 3.5天）及答辩评审标准，本版本（V4.0）对迭代二的功能域进行了重大战略调整：

1. **AI 功能必做化**：原选做的 AI 辅助策划、AI 敏感词前置过滤、AI 图片分类全面转为“迭代二必做”，并进行轻量化接口设计，确保 100% 交付。
2. **粒度粗化与控制 Scope**：为了缓解团队交付压力，对迭代二冗余及相近的业务故事进行了深度“归类合并”，同时将 IM 消息已读/撤回、小队多级权限、自动化积分榜等高复杂度故事降级为“选做/后期演进”。
3. **精准控制规模**：成功将迭代二必做总数（含 4 个全期质量保障 NFR）精准控制在 20 个，全面释放团队的研发与测试吞吐量。

## 二、 术语表 (Glossary)

- **MVP (Minimum Viable Product，最小可行性产品)**：指具备支撑业务主骨架运行的、包含最少核心功能的版本。本次指迭代一交付的“纯文本自闭环底座版”。
- **用户故事 (User Story, US)**：从用户的角度出发，对系统功能和价值的一种轻量化描述。格式为“作为一个...我希望...以便...”。
- **验收标准 (Acceptance Criteria, AC)**：针对单个用户故事设定的边界和通过条件，是研发编码与 QA 测试的唯一基准。
- **NFR (Non-Functional Requirement，非功能性需求)**：指系统在性能、安全性、可靠性、兼容性等技术层面的硬性指标约束。

## 五、 版本发布与交付里程碑

### 4.1 交付生命周期路线图

- **初始阶段 (6月27日-6月28日)**：需求分析，划分故事，完成 V4.0 关键需求文档评审。
- **第一次迭代 (6月29日-7月1日)**：完成故事细化任务，开展迭代开发，进行纯文本主链路集成测试，并于 7月1日下午进行产品评审（交付可运行纯文本 MVP 版）。
- **第二次迭代 (7月2日-7月6日)**：高级地理空间扩展、AI 智能层、核心社交链路开发，进行 NFR 高并发性能加固，并于 7月6日下午进行组间评审互评（全量交付）。

### 4.2 迭代准入与准出硬性克制阀门

- **迭代一准出阀门**：13个基础用户故事 100% 通过功能性测试，不包含地图、AI、IM等组件，可以不依赖任何未来代码实现“用户登录、发布纯文本活动、首页搜到该活动”的业务流闭环。
- **迭代二准出阀门**：20个必做故事完全通过黑盒及非功能性能校验，大模型 API 交互稳定。

## 六、 团队资源规划与工时配比

本团队由 10 人组成（1名 PO 负责需求与验收，1名 SM 负责协调与看板进度控制，8名 Developer 划分为四个结对编程小组：A组、B组、C组、D组）。

- **迭代一可用总工时上限**：8人 × 2.5天 × 8小时 = 160工时（实际排期 70 工时，负载率 43.75%，预留极高缓冲用于填补开发环境搭建及工具链阻难）。
- **迭代二可用总工时上限**：8人 × 3.5天 × 8小时 = 224工时（实际排期 102 工时，负载率 45.54%，预留充分空间解决第三方高德地图 SDK 挂接及 AI 接口异步联调延迟）。

## 七、 测试与全期质量保障规划

1. **单元测试规范 (Unit Test)**：开发人员在编写核心业务逻辑（如 JWT 签发、密码加密、状态机转换）时，必须编写 JUnit / PyTest 断言，核心 API 逻辑覆盖率达到 80% 以上。
2. **每日冒烟测试 (Smoke Testing)**：每日傍晚代码合并至主分支后，由 Scrum Master 执行最核心的主链路测试（如：登录、发帖、报名），确保主干代码编译不发生崩溃。
3. **集成与全链路系统测试**：
   - 迭代一末尾：集中验证“用户注册、JWT登录、录入文本发布活动、首页刷新可见”这一纯文本闭环链路的畅通性。
   - 迭代二末尾：执行高强度的交叉与并发链路测试，利用 JMeter 模拟多人并发报名抢单，压力测试 Redis 分布式锁，确保系统不超卖。
